## 什么是 Litestar？

<HoverDetail term="Litestar" href="https://litestar.dev">Litestar（前身 Starlite）是一个生产级的、异步优先的 ASGI Python Web 框架，在设计上追求"异步优先但同步代码零惩罚"，默认使用 Rust 加速的 msgspec 做数据验证和序列化，在多数基准测试中吞吐量优于 FastAPI。GitHub 约 5,900+ stars（2026）。</HoverDetail> 是一款轻量但全功能的 <HoverDetail term="ASGI" href="https://asgi.readthedocs.io">ASGI（Asynchronous Server Gateway Interface）是 Python Web 服务器与异步 Web 框架之间的标准接口，是 WSGI 的异步继任者，支持 WebSocket、HTTP/2 和长连接等现代协议。</HoverDetail> Python 框架。它的前身是 **Starlite**（因与 Starlette 名称过于相似而改名），核心设计哲学是：

> **"异步优先，但同步代码零惩罚。"**

与其他全栈框架不同，Litestar 不绑手绑脚——你可以从简单 API 起步，逐步扩展到大型分层项目，框架能力随需求自然生长。

## 性能基准

Litestar 的杀手锏是默认使用 <HoverDetail term="msgspec" href="https://jcristharif.com/msgspec/">msgspec 是一个基于 Rust 的 Python 序列化库（通过 PyO3），提供 JSON/MessagePack 编码解码和 Schema 验证，比 Pydantic v2 快约 12 倍，比标准 json 快约 85 倍。Litestar 用它替代 Pydantic 作为默认数据层。</HoverDetail> 做序列化——这是一套 C（Rust 通过 PyO3）实现的数据验证引擎，比 Pydantic v2 快约 **12 倍**。

## 内置能力清单

Litestar 的一大优势是**电池含量高**——以下是框架直接内置的功能：

| 类别 | 内置能力 |
|------|---------|
| **数据层** | msgspec 序列化、Pydantic v1/v2、dataclasses、attrs 多后端共存 |
| **依赖注入** | pytest 风格、分层作用域（App → Router → Controller → Handler） |
| **API 文档** | OpenAPI 3.1 自动生成 + Swagger / ReDoc / Stoplight Elements |
| **安全** | CORS、CSRF、JWT 认证、Rate Limiting、Guards 守卫机制 |
| **数据库** | SQLAlchemy 插件（含 DTO 自动映射）、原生 Repository 模式 |
| **通信** | WebSocket、GraphQL、gRPC（原生支持，无需第三方插件） |
| **中间件** | 压缩（GZip/Brotli）、Session、CORS、Logging |
| **生命周期** | `before_send` / `after_response` 细粒度钩子 |

## 代码示例

### Hello World

```python
from litestar import Litestar, get

@get("/")
async def hello_world() -> dict[str, str]:
    """Keeping the tradition alive with hello world."""
    return {"hello": "world"}

app = Litestar(route_handlers=[hello_world])
```

### 带数据验证的完整示例

```python
from litestar import Litestar, post
from msgspec import Struct

# ---- 数据模型 ----
class ItemCreateDTO(Struct):
    name: str
    price: float
    tags: list[str] = []

# ---- 路由 ----
@post("/items")
async def create_item(data: ItemCreateDTO) -> ItemCreateDTO:
    """请求体自动通过 msgspec 验证和反序列化。"""
    return data  # 自动序列化为 JSON 响应

app = Litestar(route_handlers=[create_item])
```

### 类式 Controller 组织

```python
from litestar import Controller, get, post
from litestar.dto import DTOConfig
from msgspec import Struct

class UserDTO(Struct):
    id: int
    name: str
    email: str

class UserController(Controller):
    path = "/users"

    @get("/")
    async def list_users(self) -> list[UserDTO]:
        ...

    @post("/")
    async def create_user(self, data: UserDTO) -> UserDTO:
        ...
```

## Litestar vs FastAPI：选型决策

<HoverDetail term="依赖注入（DI）" href="https://docs.litestar.dev/latest/usage/dependency-injection.html">Litestar 的依赖注入系统借鉴了 pytest 的 fixture 设计，通过 Provide 函数声明依赖，支持 App、Router、Controller、Handler 四级作用域。FastAPI 的 Depends() 更隐式，缺少层级化的生命周期管理。</HoverDetail> 是两者最显著的架构差异之一。

| 场景 | 推荐框架 | 理由 |
|------|---------|------|
| 高并发微服务 | **Litestar** | msgspec 序列化快 12×，内存占用低 50% |
| 团队已熟悉 FastAPI | FastAPI | 降低重培训成本 |
| 需要 GraphQL/gRPC | **Litestar** | 原生支持，无需第三方 |
| 大量第三方集成 | FastAPI | 生态大 10 倍以上 |
| 新项目，追求性能 | **Litestar** | 技术债更少，内置安全功能更全 |
| 招聘优先 | FastAPI | 候选人池大（OpenAI、Uber 等都在用） |

> **共识**：框架吞吐量差异在真实数据库负载下会大幅缩小。选型应优先考虑团队经验、生态需求和长期维护——而非微基准。

## 核心设计亮点

1. **DTO 系统**：通过插件架构，SQLAlchemy 模型可以直接作为路由参数类型，框架自动处理序列化和 OpenAPI Schema 生成——无需手写中间转换层。

2. **Guards 守卫**：路由级别的鉴权守卫，支持组合和优先级，比 FastAPI 的 `Depends` 注入更显式、更安全。

3. **分层架构**：原生支持 Controller → Service → Repository 分层组织，对于大型项目来说，比 FastAPI 的扁平路由文件更易维护。
