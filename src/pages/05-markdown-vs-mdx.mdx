## 一句话结论

> **纯文字、图片、代码块、表格 → Markdown。需要可复用组件、可视化、数据驱动渲染 → MDX。**

<HoverDetail term="MDX" href="https://mdxjs.com">MDX 是 Markdown + JSX 的混合格式，允许在 Markdown 文档中直接使用 React/Vue 等前端组件。它与 Markdown 的关系不是"高级 vs 低级"，而是"静态文档 vs 交互应用"——选择权在作者手中。</HoverDetail> 不是"更高级"的 Markdown——它只是让内容成为应用的一部分。

但**这篇文章本身就是一个证明**：如果一篇讲"组件优势"的文章自己不用组件，那说服力为零。下面的四个场景对比，你会看到同一个信息在两种格式下的表达力差异。

## 四个场景：用组件展示组件的价值

### 场景一：普通技术段落

如果只是写一段介绍性文字——没有交互、没有数据、没有结构对比——**两者完全相同**。

> 这个段落本身就是一个例子。你正在读的这段文字，用 Markdown 或 MDX 写出来没有任何区别。当内容不需要"组件级别的表达力"时，选择最简单的工具。

**结论**：纯文字场景 Markdown 与 MDX 等价。**不要为了用 MDX 而用 MDX。**

### 场景二：Pros/Cons 分析卡片

当你展示对比信息时，区别立刻显现。

<HoverDetail term="组件复用" href="https://react.dev/learn/thinking-in-react">组件复用的核心价值：定义一次，到处使用。当团队需要写 10 篇 SWOT 分析时，组件方案只需要维护一处样式；Markdown 需要维护 10 处手写排版。这不仅是效率问题——更是设计一致性的保障。</HoverDetail> 带来的不只是"好看"，而是**团队协作中视觉一致性**的根本保障。

### 场景四：数据驱动渲染

当内容依赖于数据时，MDX 的优势不可替代。

<DataChart
  data={[

    { method: "MDX", 维护成本: 2, 修改灵活性: 9, 信息密度: 9 },
  ]}
  type="bar"

  xKey="method"
  color="#f59e0b"
  title="教程步骤维护方式对比（分数越低维护成本越低）"
  height={240}
/>

Markdown 写教程步骤：逐条手动编写，每改一步需要手动调整所有序号。MDX：对数组执行 `map()`，数据变，内容自动变。这就是 <HoverDetail term="数据驱动渲染" href="https://react.dev/learn/rendering-lists">数据驱动渲染（Data-Driven Rendering）是 React 的核心范式：UI 是数据的函数。当数据变化时，UI 自动更新。在 MDX 中，这意味着教程步骤、参数矩阵、变更日志等结构化内容可以定义为数据数组，由组件自动生成——零人工维护成本。</HoverDetail> 的威力。

### 一句话判断

| 内容感觉像... | 用什么 |
|--------------|--------|
| **一篇文章** | Markdown |
| **产品界面的一部分** | MDX |
| **需要可复用的结构化表达** | MDX |
| **纯参考/记录** | Markdown |

## 这篇文章用了哪些组件？

既然文章主题是"用组件展示组件的价值"，这篇文章自己就用到了：

| 组件 | 用途 | 对应场景 |
|------|------|---------|
| `DataChart` ×3 | 决策框架可视化 | 场景四（数据驱动） |
| `SmartImageCard` ×4 | A vs B 对比卡片 | 场景二（结构化对比） |
| `HoverDetail` ×4 | 术语即时解释 | 全文（减少认知摩擦） |
| `ArticleTag` ×4 | 文章分类标签 | 知识管理 |
