## 先给结论（30 秒版本）

- 内容是**纯文字、图片、代码块、表格** → **Markdown**
- 需要**复用组件、插入可视化、条件/循环渲染** → **MDX**

不是"MDX 一定更高级"，而是——

> **"MDX 让内容变成应用的一部分。"**

---

## 示例 1：普通技术段落（两者完全一样）

**Markdown：**

```markdown
## 什么是缓存击穿

缓存击穿是指热点 key 在失效瞬间，大量请求直接打到数据库。

### 常见缓解方式

1. 热点 key 永不过期 + 主动刷新
2. 互斥锁
3. 多级缓存
```

**MDX：**

```mdx
## 什么是缓存击穿

缓存击穿是指热点 key 在失效瞬间，大量请求直接打到数据库。

### 常见缓解方式

1. 热点 key 永不过期 + 主动刷新
2. 互斥锁
3. 多级缓存
```

> **两者完全相同。**没有交互、组件、动态数据时，用 Markdown——最稳定，没有额外的构建复杂度。

## 示例 2：优缺点分析卡片（MDX 开始体现价值）

同一个分析内容，两种表达方式。

**Markdown 写法**（只能写成列表）：

```markdown
### 优点
- 开发效率高
- 社区成熟

### 缺点
- 包体积增长快
- 架构边界容易失控
```

**MDX 写法**（用组件承载同一份信息）：

```mdx
import SwotCard from "../components/SwotCard.astro";

<SwotCard
  title="使用 MDX 写技术内容"
  strengths='[{"label":"复用组件","desc":"图表、卡片、流程块一次开发，多篇复用"},{"label":"表达力强","desc":"一篇文章可同时包含文档与交互"}]'
  weaknesses='[{"label":"复杂度上升","desc":"构建链路与运行时依赖更重"},{"label":"编辑门槛提高","desc":"作者需要理解 JSX 基础"}]'
/>
```

**MDX 实际渲染效果：**

<SwotCard
  title="使用 MDX 写技术内容"
  strengths='[{"label":"复用组件","desc":"图表、卡片、流程块一次开发，多篇复用"},{"label":"表达力强","desc":"一篇文章可同时包含文档与交互"}]'
  weaknesses='[{"label":"复杂度上升","desc":"构建链路与运行时依赖更重"},{"label":"编辑门槛提高","desc":"作者需要理解 JSX 基础"}]'
/>

> **Markdown 只能表达"内容本身"；MDX 可以把"展示组件"纳入内容系统。**`SwotCard` 定义一次，全站所有分析类文章统一视觉——这就是 <HoverDetail term="组件复用" href="https://react.dev/learn/thinking-in-react">组件复用的本质：定义一次样式和结构，之后每次只需要传入数据。10 篇 SWOT 文章 = 1 处样式维护点，而不是 10 处手写排版。</HoverDetail> 的价值。

## 示例 3：流程展示（Markdown 是列表，MDX 是可视化）

**Markdown 写法：**

```markdown
## 发布流程

1. 写完正文
2. 本地预览
3. 运行 lint 与 build
4. 发布上线
```

**MDX 写法：**

```mdx
import StepFlow from "../components/StepFlow.astro";

<StepFlow
  title="博客发布流程"
  itemsJson='[{"step":1,"label":"写完正文"},{"step":2,"label":"本地预览"},{"step":3,"label":"运行 lint 与 build"},{"step":4,"label":"发布上线"}]'
/>
```

**MDX 实际渲染效果：**

<StepFlow
  title="博客发布流程"
  itemsJson='[{"step":1,"label":"写完正文"},{"step":2,"label":"本地预览"},{"step":3,"label":"运行 lint 与 build"},{"step":4,"label":"发布上线"}]'
/>

> **Markdown 的流程是"静态列表"；MDX 把流程转成结构化可视化。**同样的四步，组件的<HoverDetail term="信息密度" href="https://en.wikipedia.org/wiki/Data_density">信息密度（Information Density）是单位面积内传达的有效信息量。StepFlow 用编号圆 + 箭头连线，读者一眼看到全流程，认知负荷远低于逐行阅读 1-2-3-4 列表。</HoverDetail> 远超文字列表。

## 示例 4：数据驱动渲染（Markdown 手写，MDX 自动生成）

**Markdown 写法**（每行手动维护）：

```markdown
## 发布前检查清单

- [x] Front matter 完整
- [x] 标题层级正确
- [ ] 外链可访问
- [ ] 图片已压缩
```

**MDX 写法**（数据驱动，自动生成列表）：

```mdx
## 发布前检查清单

<ul>
  {[
    { name: 'Front matter 完整', done: true },
    { name: '标题层级正确', done: true },
    { name: '外链可访问', done: false },
    { name: '图片已压缩', done: false },
  ].map((item) => (
    <li key={item.name}>
      [{item.done ? 'x' : ' '}] {item.name}
    </li>
  ))}
</ul>
```

**MDX 实际渲染效果：**

    <li class="flex items-center gap-2">

> **Markdown 需要手写每一行，增删条目 = 手动改序号。MDX 基于数据生成内容**——尤其适合教程步骤、参数矩阵、版本变更列表。数据变，内容自动变，零人工维护。

## 决策框架

| 场景 | 建议 |
|------|------|
| 个人博客、日常技术记录、README | **Markdown 优先** |
| 文档站需要统一卡片/告警/流程组件 | **MDX 更合适** |
| 交互式教程、可视化内容、动态渲染 | **MDX 必选** |
| 多人协作且作者不懂 JSX | **Markdown 更稳妥** |

### 最后一句话

> **内容像"文章"，用 Markdown。内容像"产品界面的一部分"，用 MDX。**

不要为了"看起来高级"强行上 MDX。只有当**组件复用和动态表达真的带来收益**时，MDX 才值得。

## 这篇文章本身就是证明

你刚刚看到的 `SwotCard` 和 `StepFlow` 组件，是专门为这篇文章创建的——但它们现在是**全站可复用的资产**。以后任何文章需要做 SWOT 分析或展示步骤流程，直接导入即可，一行组件代码搞定。

这就是 MDX 的核心价值：**每次写作都在为下一次写作积累可复用的表达力。**
